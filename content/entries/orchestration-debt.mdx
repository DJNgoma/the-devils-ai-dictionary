devilDefinition: |
  The accumulated complexity produced by adding one more step, one more model, one more fallback, and one more apology until nobody can explain the workflow without a guided tour.
plainDefinition: |
  Orchestration debt is the maintenance burden created when an AI workflow accumulates too many steps, branches, models, tools, retries, and special cases for the team to reason about confidently.
whyExists: |
  Useful systems rarely stay simple. Teams add retrieval, validators, fallbacks, routing rules, approvals, tool calls, and recovery logic incrementally. The term exists because those sensible additions can eventually produce a coordination layer more fragile and opaque than the problem it was meant to solve.
misuse: |
  It should not be used to insult every non-trivial workflow. Some orchestration is necessary. It becomes debt when the layering makes failures harder to predict, ownership harder to assign, and changes harder to ship without accidentally summoning three side effects and an incident review.
practicalMeaning: |
  In practice it usually means duplicated logic, hidden dependencies, branching nobody wants to touch, unclear state transitions, and an architecture whose strongest testing strategy is hoping the person who built it is still employed.

Orchestration debt is what happens when a clever workflow wins too many arguments against simplification. The system still works, but each new improvement must now travel through customs.
