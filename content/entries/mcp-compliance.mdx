  The moment a protocol badge is asked to do the emotional labour of a security programme.
plainDefinition: |
  MCP compliance is a claim that a tool, server, client, or platform follows the Model Context Protocol or a related set of compatibility requirements.
whyExists: |
  Protocols need conformance language so implementations can interoperate and buyers can understand which pieces fit together. Compliance can clarify supported capabilities, message shapes, transport expectations, and compatibility with clients.
misuse: |
  It gets stretched into a trust claim. Supporting MCP does not prove a server is secure, well governed, maintained, private, least-privilege, or appropriate for production. It mostly says the plumbing can speak the dialect.
practicalMeaning: |
  In practice it means separating protocol conformance from organisational controls: authentication, permission review, logging, vendor risk, data handling, version pinning, incident response, and the tedious work a badge cannot perform.
example: |
  "The server was MCP compliant, which explained why the client could call it, not why anyone had allowed it access to payroll."

Compliance language is useful when it stays in its lane. The trouble begins when it is promoted from "speaks protocol" to "safe to buy" without doing the intervening paperwork.
