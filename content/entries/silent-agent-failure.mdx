plainDefinition: |
  A silent agent failure occurs when an agent completes without an infrastructure error but produces an incorrect, incomplete, unauthorised, or otherwise unacceptable outcome. The trace looks operationally healthy until someone evaluates what the agent actually did.
whyExists: |
  Conventional monitoring catches crashes, latency spikes, and failed tool calls. Agents can instead skip a prerequisite, claim an action succeeded when it did not, or continue after a tool returned unusable data. Behavioural evaluation is needed because process health and task correctness are different measurements.
misuse: |
  The phrase is sometimes used for every disappointing answer. A silent failure is more specific: the system's ordinary success signals remain green while the intended outcome is wrong. A visible exception, refusal, or timeout may still be a failure, but it is not silent in this sense.
practicalMeaning: |
  Define outcome-level checks, required steps, policy constraints, and escalation conditions. Sample completed sessions, compare traces with those expectations, cluster recurring failure patterns, and make customer complaints an input rather than the primary monitoring system.
