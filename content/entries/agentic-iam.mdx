  Identity and access management after the software stops merely requesting access and starts making plans with it.
plainDefinition: |
  Agentic IAM is identity and access management adapted for AI agents that can use tools, delegate tasks, act across systems, and make repeated decisions under scoped authority.
whyExists: |
  Traditional IAM already handles users, groups, services, roles, and policies. Agentic systems add non-human actors with goals, tool choices, delegated user authority, temporary tasks, run-level context, and side effects that need attribution and control.
misuse: |
  It gets reduced to giving the agent a service account. That may create a credential, but it does not answer who authorised a specific action, which task justified it, how long the authority should last, or what policy stopped the agent from becoming enthusiastic in the wrong system.
practicalMeaning: |
  In practice it means agent identities, delegated authentication, just-in-time scopes, policy engines, approval gates, run-level audit trails, credential expiry, revocation, and access reviews that understand software can now be both caller and decision-maker.
example: |
  "Agentic IAM arrived after the support agent used a perfectly valid token for a task no one had meant to authorise."
