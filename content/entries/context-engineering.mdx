  The rebranding of prompt fiddling once people realised the prompt had quietly become half the product.
plainDefinition: |
  "Context engineering" is the practice of deciding what instructions, tools, retrieved information, memory, examples, and constraints a model should receive so it can perform a task reliably.
whyExists: |
  It exists because model quality depends heavily on what reaches the context window and in what order. Once systems grew beyond a single prompt box, teams needed a name for the work of shaping that information flow deliberately.
misuse: |
  It gets inflated into a grand discipline whenever someone wants ordinary application design to sound frontier and proprietary. Sometimes it really is nuanced systems work. Sometimes it is just a long string and a lucky example.
practicalMeaning: |
  In practice it means prompt templates, retrieval rules, truncation policies, memory selection, tool schemas, system instructions, and a permanent struggle over what the model should see before it starts improvising.
example: |
  "The context engineering improved dramatically once they stopped stuffing every policy document into the prompt and calling it judgment."
