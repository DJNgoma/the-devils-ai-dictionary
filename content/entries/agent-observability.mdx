devilDefinition: |
  The art of discovering what the autonomous system did after everyone has finished admiring that it did something.
plainDefinition: |
  Agent observability is the ability to inspect an agent workflow across model calls, tool calls, state changes, approvals, errors, costs, and outcomes.
whyExists: |
  Agent workflows fail across several layers at once. A bad result may come from the model, the prompt, a tool, stale context, permission design, retry logic, or the handoff between them. Observability exists because a final answer is not enough evidence to debug the route that produced it.
misuse: |
  It gets reduced to pretty traces or token charts. A dashboard is not observability if it cannot show what happened, what mattered, who approved it, which tool changed state, and where recovery was still possible.
practicalMeaning: |
  In practice it usually means run traces, tool-call logs, prompt and context snapshots, cost breakdowns, approval records, replayable failures, and enough structure for operations to tell whether the agent completed the work or merely produced a confident artefact.
example: |
  "The agent observability review found that the model was not confused until the tool returned yesterday's account data with today's schema."
