  The point where the model stops guessing alone and starts borrowing other systems' competence.
plainDefinition: |
  Tool use is the pattern of giving a model access to external systems such as search, code execution, databases, browsers, calculators, or internal APIs so it can gather information or take actions.
whyExists: |
  Because models know many things badly, some things approximately, and almost nothing about the current state of your systems unless another tool tells them.
misuse: |
  By marketing tool access as autonomy rather than integration. The hard part is not that the model can ask for a tool. The hard part is deciding which tools it may touch, when, and under whose supervision.
practicalMeaning: |
  In practice it means schemas, permission scopes, retries, observability, error handling, and a constant negotiation between flexibility and the desire to avoid discovering new incident classes at scale.

  "The assistant became materially more useful once tool use let it search the docs, check the account, and stop inventing the answer from memory."
