devilDefinition: |
  The architectural admission that reading the prompt and inventing the next token are different jobs, even when marketing still sells them as one smooth thought.
plainDefinition: |
  Prefill/decode split is a serving architecture pattern that separates the compute-heavy processing of the input prompt (prefill) from the sequential generation of output tokens (decode), often so each phase can be scheduled, batched, cached, or hardware-tuned differently.
whyExists: |
  Prompt processing is typically parallel and memory-bandwidth hungry; token generation is sequential and latency-sensitive. Treating them as one uniform workload wastes GPUs, confuses capacity planning, and makes "tokens per second" sound more honest than it is.
misuse: |
  It is treated as a guarantee of lower cost or magic speed. A split only helps when the scheduler, KV cache, batching, and hardware placement actually exploit the different shapes of work. Naming the phases is not the same as running them well.
practicalMeaning: |
  In practice it means sizing and routing prefill and decode separately, watching TTFT versus inter-token latency, deciding where KV state lives, and refusing to judge the whole stack by a single tokens-per-second slide.
