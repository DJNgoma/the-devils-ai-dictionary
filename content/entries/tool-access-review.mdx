devilDefinition: |
  The meeting where the tool list is supposed to become a control surface, not a souvenir from the prototype.
plainDefinition: |
  A tool access review is a recurring or pre-release review of which tools an AI system can use, what each tool can do, who owns the access, and whether the permissions still match the workflow's risk.
whyExists: |
  Tool access drifts. Prototypes gain connectors, pilots inherit credentials, teams add exceptions, and the original permission story quietly becomes folklore. Review exists to compare the declared design with the live system.
misuse: |
  It is reduced to asking whether the team feels comfortable with the current setup. Comfort is not evidence. The review has to inspect real grants, scopes, logs, owners, and revoke paths.
practicalMeaning: |
  In practice it means inventorying live connectors, checking read and write scopes, confirming user impersonation rules, sampling audit logs, removing unused tools, and giving somebody accountable the power to say no before the agent says "done".
example: |
  "The tool access review found that the calendar agent still had a file-export scope from a prototype nobody wanted to remember."
