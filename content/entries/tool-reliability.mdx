devilDefinition: |
  The unfashionable question of whether the supposedly helpful thing the model keeps calling will actually behave the same way twice.
plainDefinition: |
  Tool reliability is the consistency, correctness, and recoverability of a tool when invoked by a model or agent under normal and adverse operating conditions.
whyExists: |
  An agent is only as dependable as the tools it calls. Even strong reasoning degrades rapidly when the surrounding tools are flaky, under-specified, slow, or capricious about their outputs.
misuse: |
  It gets reduced to uptime or successful HTTP responses. A tool can be "up" and still be unreliable because it returns partial state, inconsistent schemas, non-idempotent side effects, or errors too vague for the caller to recover from intelligently.
practicalMeaning: |
  In practice it usually means schema stability, timeouts, retries, idempotency, versioning, error semantics, sandbox boundaries, and the quiet labour of making external actions predictable enough that a model can call them without inventing theology around transient failure.
