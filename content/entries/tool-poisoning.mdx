devilDefinition: |
  Hiding instructions in the tool layer so the agent mistakes a compromised capability for a helpful one.
plainDefinition: |
  Tool poisoning is an attack or failure mode in which a tool's descriptors, schemas, metadata, routing, or surrounding instructions are manipulated so an agent invokes it under false or malicious assumptions.
whyExists: |
  Modern agents do not merely call tools; they read about them. If the model relies on tool descriptions to decide what is safe, relevant, or authoritative, poisoning that layer becomes a prompt injection problem with better infrastructure.
misuse: |
  It should not be used for every bad tool output. The more precise issue is corruption of the tool interface itself: the agent is misled about what the tool is, what it does, or how it should be used.
practicalMeaning: |
  In practice it means malicious MCP descriptors, misleading plugin metadata, tampered schemas, or tool naming collisions that redirect the agent into unsafe actions, data leakage, or expensive nonsense.
example: |
  "The agent did not get hacked by sorcery; it was simply handed a poisoned tool description and obeyed it with enterprise enthusiasm."
