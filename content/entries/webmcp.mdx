  The web discovering that if agents are going to click around anyway, it may as well leave them a labelled door.
plainDefinition: |
  WebMCP is a protocol-adjacent idea for exposing website, browser, or web application capabilities to agents through MCP-style discovery, resources, tools, or structured actions.
whyExists: |
  Browser agents can already inspect pages and perform actions, but ordinary web interfaces are built for humans. A web-facing MCP pattern can make capabilities more explicit, typed, permissioned, and less dependent on brittle clicking.
misuse: |
  It gets framed as the end of messy web automation. It is not. Standardised exposure still has to handle identity, consent, rate limits, abuse controls, layout changes, commercial incentives, and the fact that websites may not want agents using them like private APIs.
practicalMeaning: |
  In practice it means websites deciding which actions and resources they expose to agents, how users authorise them, what clients can discover, and where the boundary sits between helpful automation and scraping with better manners.
example: |
  "WebMCP let the travel site expose a booking search cleanly, which was preferable to watching an agent interpret three pop-ups and a loyalty-programme ambush as interface design."

The web has spent decades tolerating automation through side doors. WebMCP is the argument for making some of those doors official, lockable, and auditable.
